Case Study

ChairTime

A real booking system, not just a landing page

Self-directed portfolio case study based on a hypothetical brief — not a paid client engagement. No fabricated testimonials.
Niche
Barbershop & salon booking
Type
Booking platform (client + staff + admin)
Role
Solo design & development
Stack
React, TypeScript, Vite, Zustand
Live demo GitHub Original template: Blackwood Barbershop

Context

Blackwood Barbershop already exists in this portfolio as a template landing page. ChairTime asks: what does this same shop need once a static page stops being enough?

Challenge

A landing page can list hours and services. It can't let four barbers manage independent schedules, let a client book a specific barber at a specific time, or give the owner a single view of the whole shop's day. That's a different category of software.

Approach

Three roles, one shared data model — a client booking flow, a per-barber staff dashboard, and a shop-wide admin dashboard, all reading from the same appointment state. Booking availability is computed live from each barber's working hours and existing appointments, not hardcoded.

Solution

  • 4-step booking wizard with a radial time picker, with a full keyboard/list-view fallback for accessibility
  • Role-based dashboards: Client / Staff / Admin, switchable in a demo mode labeled honestly as a demo
  • Admin can manage services, staff working hours, and any booking — changes reflect in client-facing availability immediately
  • Bilingual EN/RU, persisted state via localStorage

Results

Designed-for outcomes — no real traffic to report.

  • Reduces booking from a phone call to a ~90-second flow
  • Eliminates double-booking via per-staff availability locking

What I'd build next

A real backend (Supabase/Firebase), real SMS/email confirmations, payment collection for deposits.
