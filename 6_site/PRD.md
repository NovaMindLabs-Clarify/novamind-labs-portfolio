# PRD — «ChairTime»: платформа онлайн-записи для Blackwood Barbershop

*Premium-проект №1 (см. `PORTFOLIO_STRATEGY_PLAN.md`, раздел 3). Апгрейд ниши барбершопа из `2_site` — та же студия Blackwood, но с полноценной системой записи вместо статичного лендинга. Документ объединяет и PLAN, и PRD в одном файле (в отличие от тира 1) — для premium-проектов это единственный источник правды по продукту, максимально подробный по всем направлениям, как и просил заказчик.*

---

## 1. Обзор и цель проекта

**Что это:** веб-приложение онлайн-записи для барбершопа с тремя ролями — клиент, барбер (сотрудник), администратор/владелец. Не лендинг с формой-заглушкой, а реальная логика: занятость по мастерам, статусы записи, управление услугами и расписанием.

**Зачем в портфолио:** это флагманский premium-кейс, доказывающий, что автор умеет проектировать и реализовывать многоролевые интерфейсы с состоянием, а не только статичную вёрстку. Прямая связка с `2_site` («тот же барбершоп, но теперь с настоящей системой записи») делает историю понятной клиенту без дополнительных объяснений.

**Продуктовая рамка (честно, без обмана посетителя портфолио):** это самостоятельный демонстрационный проект по гипотетическому ТЗ, не реальный работающий бизнес и не оплаченный заказ. Никаких выдуманных отзывов реальных клиентов Blackwood — только спроектированные (не вымышленно-статистические) результаты и обозначение «портфолио-кейс» в самом приложении и в будущей странице кейс-стади.

**Основная цель конверсии (для посетителя портфолио):** посетитель должен за 2–3 минуты поочерёдно попробовать все три роли (клиент → бронирует, барбер → видит свой день, админ → видит доску на весь зал) и прийти к выводу «этот человек умеет в реальные интерфейсы, не только в лендинги».

## 2. Границы проекта

**В скоуп:**
- Полностью клиентское SPA-приложение (React), без реального бэкенда — «база данных» это мок-данные + `localStorage`, аналогично тому, как формы в тире 1 симулируют отправку.
- Три ролевых интерфейса в одном приложении с переключателем ролей (демо-режим — без настоящей авторизации, это заявлено явно в самом UI).
- Публичный флоу бронирования (4 шага), дашборд барбера (свой день/неделя), дашборд администратора (весь зал, управление услугами/сотрудниками/бронированиями, базовая статистика).
- Двуязычность EN (основной) / RU — сохраняем конвенцию всего портфолио.
- Адаптивная вёрстка 320–1920px.

**Вне скоупа:**
- Реальная авторизация/бэкенд/база данных, реальные платежи, реальные email/SMS-уведомления (email/SMS обозначаются в UI как «будет отправлено», но фактически не отправляются — это отмечается в разделе «Что добавил бы для продакшена» будущей кейс-стади).
- Мультиарендность (multi-tenant, несколько барбершопов) — приложение жёстко сконфигурировано под один демо-барбершоп (Blackwood).
- Нативные мобильные приложения.

## 3. Роли пользователей

Приложение открывается с явным демо-переключателем ролей в шапке (`Viewing as: Client / Staff / Admin`) — это заменяет реальную авторизацию и одновременно честно сигнализирует посетителю портфолио, что перед ним демо, а не продакшен-логин.

| Действие | Client (клиент) | Staff (барбер) | Admin (владелец/админ) |
|---|---|---|---|
| Забронировать стрижку | ✅ | — | ✅ (от лица клиента, ручной ввод) |
| Посмотреть свой день/неделю | — | ✅ (только свой график) | ✅ (все барберы) |
| Отметить статус записи (пришёл/выполнено/неявка) | — | ✅ (только свои записи) | ✅ (любые записи) |
| Отменить/перенести запись | ✅ (свою, по коду брони) | ✅ (свои, в пределах дня) | ✅ (любую) |
| Управлять услугами (цена/длительность/добавить/удалить) | — | — | ✅ |
| Управлять барберами (часы работы, выходные, добавить/удалить) | — | — | ✅ |
| Смотреть статистику зала (записи сегодня/за неделю, загрузка, no-show) | — | — | ✅ |

При переключении на Staff дополнительно нужно выбрать, за какого барбера смотреть демо (т.к. барберов четверо) — простой селектор «Which barber are you?» сразу после переключения роли.

## 4. Информационная архитектура (маршруты)

```
/                → App shell: мини-хиро "Blackwood — Book Your Chair",
                   переключатель ролей, вход в один из трёх флоу
/book            → Клиентский флоу бронирования (4 шага, см. раздел 6.1)
/book/confirmation/:code → Экран подтверждения с "билетом"
/manage/:code    → Поиск своей брони по коду для отмены/переноса (клиент)
/staff           → Выбор барбера (для демо) → дашборд барбера
/staff/:barberId → Дашборд барбера (день/неделя)
/admin           → Дашборд администратора (обзор/зал на сегодня)
/admin/services  → Управление услугами
/admin/staff     → Управление барберами и часами работы
/admin/bookings  → Управление всеми бронированиями (поиск/фильтр)
```

React Router v6, вложенные роуты для `/admin/*` с общим layout (сайдбар навигации внутри дашборда).

## 5. Модель данных (mock, хранится в localStorage через Zustand persist)

```ts
type Service = {
  id: string;
  name: { en: string; ru: string };
  durationMin: number;
  price: number;           // USD, как и в 2_site
  category: 'hair' | 'beard' | 'combo' | 'shave' | 'kids';
};

type Staff = {
  id: string;
  name: string;             // Marcus Reid, Diego Alvarez, Theo Brooks, Sam Whitaker
  role: { en: string; ru: string }; // напр. "Master Barber"
  photo: string;             // picsum seed, как в 2_site
  workingHours: {            // по дням недели, локальное время шоурума
    [day in 'mon'|'tue'|'wed'|'thu'|'fri'|'sat'|'sun']?: { start: string; end: string };
  };
  daysOff: string[];          // ISO-даты точечных выходных
};

type Appointment = {
  id: string;
  code: string;                // короткий код брони для /manage/:code, напр. "BW-4F2K"
  serviceId: string;
  staffId: string;
  clientName: string;
  clientPhone: string;
  note?: string;
  date: string;                 // ISO YYYY-MM-DD
  startTime: string;             // "HH:mm"
  endTime: string;               // вычисляется из durationMin
  status: 'booked' | 'arrived' | 'completed' | 'no-show' | 'cancelled';
  createdAt: string;              // ISO timestamp
};
```

**Сиды при первом запуске:** 6 услуг и 4 барбера — переносятся 1:1 из `2_site/js/i18n.js` (Classic Haircut $32 / Beard Trim & Shape $20 / Cut + Beard Combo $45 / Hot Towel Shave $35 / Kids Cut $22 / Buzz Cut $25; барберы Marcus Reid, Diego Alvarez, Theo Brooks, Sam Whitaker) — это и есть видимая связь с тиром 1. Плюс 15–20 сид-записей на ближайшие 7 дней с разными статусами, чтобы дашборды не были пустыми при первом визите.

## 6. Детальные UX-флоу

### 6.1 Клиент: бронирование (4 шага + подтверждение)

**Шаг 1/4 — Выбор услуги.** Сетка карточек услуг (название, длительность, цена), сгруппированных по категориям вкладками (как меню в `2_site`, уже знакомый паттерн). Клик по карточке — выбор, кнопка «Next» активируется.

**Шаг 2/4 — Выбор барбера.** Карточки 4 барберов (фото, имя, специализация) + карточка «No preference — first available». Под каждой карточкой — теговая подсказка ближайшего свободного слота («Next available: Today, 3:30 PM»), чтобы решение о выборе барбера было информированным.

**Шаг 3/4 — Выбор времени (сигнатурный элемент, см. раздел 8.2).** Горизонтальная лента из 7 дней сверху (сегодня + 6 дней вперёд), под ней — циферблат карманных часов: доступные часы — активные насечки на латунном диске, занятые/нерабочие — визуально «спилены»/неактивны. Тап по насечке выбирает время. Для мобильных экранов и для доступности — переключатель «List view» показывает то же самое списком (см. раздел 10, доступность).

**Шаг 4/4 — Контактные данные и подтверждение.** Поля: имя, телефон, комментарий (необязательно). Справа/снизу — «корешок билета»: превью итоговой записи (услуга, барбер, дата/время, цена). Кнопка «Confirm Booking» с клиентской валидацией обязательных полей.

**Подтверждение.** Экран с анимацией «пробивки билета» (лёгкий, однократный, уважает `prefers-reduced-motion`), показывает код брони крупно, кнопки: «Add to Calendar» (генерация и скачивание `.ics`-файла — реальная работающая фича, не симуляция), «Book Another», «Manage This Booking» (ведёт на `/manage/:code`).

### 6.2 Барбер: дашборд

После выбора «за какого барбера смотреть» — вертикальная таймлайн-доска на день: каждая запись — «билет-стаб», приколотый к своему времени. Барбер может отметить запись как Arrived / Completed / No-show (меняет статус, влияет на статистику). Переключатель Today / This Week (неделя — только просмотр, без действий на будущие дни). Внизу — короткая сводка дня: «6 of 9 chairs booked».

### 6.3 Админ: дашборд

**Обзор (Overview).** «Доска зала» на сегодня — 4 колонки (по барберу), время сверху вниз, записи — билеты-стабы в своих ячейках (тот же визуальный язык, что и в дашборде барбера, но по всем сразу — это и есть флагманский «admin dashboard» вид для кейс-стади). Сверху — статистическая полоса: записей сегодня, записей за неделю, самый загруженный барбер, % неявок.

**Manage Services.** Список услуг таблицей, инлайн-редактирование цены/длительности, добавление/удаление услуги (мок-CRUD, сразу пишется в localStorage).

**Manage Staff.** Список барберов, редактирование часов работы по дням недели и точечных выходных (это напрямую влияет на доступные слоты в клиентском бронировании — важно явно показать эту связь в самом интерфейсе, например строкой «Changes affect client booking availability immediately»).

**Manage Bookings.** Таблица всех записей с фильтрами (дата, барбер, статус) и поиском по имени/телефону/коду; действия «Cancel» и «Reschedule» (открывает мини-версию шага 3 флоу бронирования).

## 7. Технический стек

- **React 18 + TypeScript + Vite** — TypeScript добавлен намеренно поверх решения «React+Vite»: для приложения с тремя ролями, статусной моделью записи и мутирующими мок-данными типизация ощутимо снижает риск ошибок и является дополнительным сигналом технической зрелости для клиента, читающего теги стека в кейс-стади.
- **React Router v6** — маршруты из раздела 4, вложенный layout для `/admin/*`.
- **Zustand + persist-middleware (localStorage)** — единое хранилище: услуги, барберы, записи, активная демо-роль. Выбран вместо Context, т.к. мутации (добавить услугу, сменить статус записи) многочисленны и нужны вне дерева компонентов (например, синхронный пересчёт занятости в дальнем компоненте циферблата).
- **CSS Modules + единый файл дизайн-токенов (`tokens.css`, CSS custom properties)** — сознательно **без** Tailwind: сигнатурный циферблат и билеты-стабы — заказные компоненты с нетривиальной геометрией (SVG-дуги, позиционирование насечек по кругу), для них выразительнее и опрятнее писать целевой CSS поверх токенов, чем комбинировать десятки utility-классов. Дизайн-токены переиспользуют часть палитры `2_site` (см. раздел 8.1) для узнаваемости бренда.
- **Framer Motion** — точечно: разовая анимация циферблата при первой загрузке шага 3, переходы между шагами мастера (горизонтальный слайд), микро-hover на карточках услуг/барберов. Никакой фоновой/декоративной анимации сверх этого — «весь бюджет смелости» уходит в один сигнатурный момент, остальное — сдержанно.
- **date-fns** — работа с датами/временем (расчёт слотов занятости, форматирование).
- **react-i18next** (или лёгкий кастомный контекст-хук, если легковеснее) — EN/RU, тот же принцип «EN по умолчанию, переключатель на RU», что и в тире 1, но идиоматично для React (не через `data-i18n`-атрибуты, а через хук перевода в компонентах).
- **Иконки** — инлайн SVG вручную (не библиотека), для полного визуального контроля, как и в тире 1.
- Сборка — статический билд Vite, деплоится как обычный статический сайт (без Node-сервера в проде).

## 8. Визуальная концепция

*Продумано осознанно под этот проект — не переиспользование стиля тира 1 «в лоб», а осмысленное развитие уже узнаваемого бренда Blackwood в более насыщенный, «премиальный» продукт.*

### 8.1 Токены дизайна

| Токен | Значение | Назначение |
|---|---|---|
| `--graphite` | `#141414` | Фон приложения (продолжает `2_site`) |
| `--charcoal-surface` | `#1F1D1B` | Поверхности карточек/дашбордов |
| `--brass` | `#C9A24B` | Основной акцент, бренд (продолжает `2_site`) |
| `--bone` | `#F2EFEA` | Текст на тёмном фоне |
| `--moss` | `#4C7A5E` | Статус "доступно / подтверждено" (новый — нужен для дашбордов, которых не было в лендинге) |
| `--rust` | `#B0503A` | Статус "занято / отменено / неявка" (новый) |

**Почему не «дефолтный AI-стиль»:** тёмный фон с латунным акцентом здесь — не случайный «near-black + один яркий акцент» шаблон, а прямое продолжение уже установленного бренда барбершопа (тот же `2_site`), плюс латунь — предметный, а не абстрактный выбор: это материал реальных инструментов барбершопа (ножницы, опасные бритвы, оправы зеркал). Два новых статусных цвета (мох/ржавчина) введены по необходимости дашбордов, а не для украшения.

### 8.2 Типографика

- **Display:** Oswald (продолжение `2_site`) — используется сдержанно: заголовки экранов, названия шагов мастера.
- **Body:** Inter (продолжение `2_site`) — весь читаемый текст, лейблы форм.
- **Utility/data (новый третий уровень для этого проекта):** моноширинный шрифт (JetBrains Mono или IBM Plex Mono) — коды бронирования, время на билетах-стабах, цифры в статистике админ-дашборда. Ассоциация с классическим билетным компостером/табло — усиливает мотив «билета», а не декоративна.

### 8.3 Сигнатурный элемент: циферблат карманных часов как выбор времени

Единственный по-настоящему смелый визуальный элемент проекта (сознательно один, не размазан по всему интерфейсу — «весь бюджет смелости в одном месте»): шаг 3 бронирования не показывает список кнопок времени, а даёт **латунный циферблат карманных часов**, где доступные часы — светящиеся насечки по кругу, а занятые — визуально «спилены». Это прямая, оправданная связь с названием проекта (ChairTime → время) и с миром барбершопа (карманные часы — узнаваемый атрибут классического барбера). Обязательно сопровождается доступной списочной альтернативой (см. раздел 10) — сигнатурный элемент не должен ломать доступность.

### 8.4 Композиция и мотив «билета»

Сквозной мотив по всему приложению (клиентское подтверждение, дашборд барбера, доска зала админа) — запись визуально представлена как **билет-стаб** (перфорированный край, моноширинный код/время) — та же визуальная единица, что и в кейсе на подтверждении бронирования у клиента, что и в ячейке на доске у админа. Повторяющийся, а не декоративный элемент — сразу считывается «это одна и та же запись», когда посетитель портфолио переключается между ролями.

### 8.5 Движение

- Разовая анимация циферблата (стрелки/насечки проявляются) при первом входе на шаг 3 — не при каждом ре-рендере.
- Горизонтальный слайд между шагами мастера бронирования.
- Лёгкий hover-подъём + латунное подчёркивание на интерактивных карточках.
- `prefers-reduced-motion: reduce` — отключает анимацию циферблата и слайды шагов (мгновенная смена вместо transition), оставляет только необходимые для UX состояния (fokus, active).

## 9. Функциональные требования по экранам (сводно)

1. **App shell (`/`)** — мини-хиро, переключатель ролей, три входные точки (Book / Staff View / Admin View), пояснение-бейдж «Portfolio demo — no real bookings are processed».
2. **Booking wizard (`/book`)** — 4 шага из 6.1, прогресс-индикатор (билеты-корешки как шаги, а не абстрактные точки — см. 8.4), возможность вернуться на предыдущий шаг без потери выбора.
3. **Confirmation (`/book/confirmation/:code`)** — билет с кодом, `.ics`-экспорт, ссылки на «забронировать ещё» и «управление бронью».
4. **Manage booking (`/manage/:code`)** — поиск по коду, показ деталей, кнопки «Cancel» / «Reschedule» (для незавершённых/будущих записей; прошедшие/завершённые — только просмотр).
5. **Staff picker (`/staff`)** — выбор барбера для демо-режима.
6. **Staff dashboard (`/staff/:barberId`)** — день/неделя, смена статусов, сводка загрузки (детали в 6.2).
7. **Admin overview (`/admin`)** — доска зала + статистическая полоса (детали в 6.3).
8. **Admin services (`/admin/services`)** — CRUD услуг.
9. **Admin staff (`/admin/staff`)** — CRUD барберов, часы/выходные.
10. **Admin bookings (`/admin/bookings`)** — поиск/фильтр/отмена/перенос по всем записям.

Каждый экран обязан: (а) корректно работать при пустых данных (пустые состояния с понятным призывом к действию — не белый экран), (б) корректно работать сразу после сброса localStorage (повторный сид), (в) быть доступен с клавиатуры полностью, без исключений для сигнатурного циферблата.

## 10. Нефункциональные требования

- **Доступность:** обязательная списочная альтернатива циферблату (переключатель «List view» / автоматический фолбэк на список при навигации клавиатурой или для screen reader — циферблат размечен как `role="radiogroup"` с доступными `aria-label` на каждой насечке, но список остаётся основным путём для не-мышиного ввода). Контраст текста ≥ WCAG AA на тёмном фоне (латунь на графите уже проверяется на контраст при подборе финального оттенка). Полная доступность с клавиатуры для всех трёх ролей, видимый focus-ring.
- **Адаптивность:** 320–1920px. На узких экранах (< 480px) циферблат заменяется вертикальным списком слотов с тем же визуальным языком (насечки → строки), а не сжимается до нечитаемого размера — это осознанное решение, не костыль.
- **Производительность:** код-сплиттинг по роутам (клиент/staff/admin грузятся отдельными чанками — нет смысла грузить админку ради клиентского бронирования), изображения барберов — те же picsum-сиды, что и в `2_site`, с `loading="lazy"` там, где не первый экран.
- **Устойчивость данных:** сид-данные пересоздаются автоматически, если localStorage пуст или повреждён (не должно приводить к белому экрану/краху).
- **Браузеры:** актуальные Chrome/Firefox/Safari/Edge, мобильный Safari/Chrome.
- **Честность демо:** везде, где в реальном продукте было бы SMS/email/оплата, интерфейс явно говорит «demo — not actually sent/charged», а не притворяется, что действие реально произошло (кроме `.ics`-экспорта, который реально работает).

## 11. Контент и локализация

Копирайт — EN основной, RU через переключатель, тот же принцип, что и в портфолио тира 1 (`localStorage`, дефолт EN). Тон — чуть более «продуктовый»/деловой, чем маркетинговый тон `2_site` (это интерфейс инструмента, а не рекламный лендинг), но сохраняет узнаваемый бренд-голос Blackwood (уверенный, без лишних слов). Названия услуг и барберов — 1:1 из `2_site/js/i18n.js`, чтобы связка тир1↔premium была буквальной, не только концептуальной.

## 12. Критерии приёмки (Definition of Done)

- [ ] Все 10 экранов из раздела 9 реализованы и наполнены контентом на EN и RU
- [ ] Демо-переключатель ролей работает, каждая роль видит ровно тот набор данных/действий, что описан в матрице раздела 3
- [ ] Полный флоу бронирования проходим от шага 1 до подтверждения без ошибок, включая рабочий экспорт `.ics`
- [ ] Циферблат корректно отражает занятость (забронированный слот недоступен для повторного выбора сразу после бронирования, без перезагрузки страницы)
- [ ] Доступная списочная альтернатива циферблату присутствует и полностью функциональна с клавиатуры
- [ ] Изменения в Admin → Manage Staff (часы работы) немедленно отражаются в доступности слотов клиентского бронирования
- [ ] Изменения в Admin → Manage Services (цена/длительность) немедленно отражаются в шаге 1 бронирования
- [ ] Данные переживают перезагрузку страницы (persist через localStorage) и корректно пересеиваются при первом запуске/очищенном хранилище
- [ ] Адаптивность 320–1920px без горизонтального скролла, циферблат имеет рабочую мобильную альтернативу
- [ ] `prefers-reduced-motion` отключает декоративную анимацию циферблата и переходов, не ломая функциональность
- [ ] Lighthouse (мобильный): Performance/Accessibility/Best Practices ≥ 90 на каждом из трёх ролевых дашбордов
- [ ] Нет ошибок в консоли браузера на всех 10 экранах и при переключении ролей
- [ ] В App shell и на будущей странице кейс-стади явно обозначено, что это портфолио-демо, а не реальный сервис Blackwood
